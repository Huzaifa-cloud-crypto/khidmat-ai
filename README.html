    Matching Factors (10-Factor System)

    backend/src/agents/matchingAgent.js scores every candidate provider using 10 factors:

    #FactorMax PointsLogic

    1Distance+30max(0, 30 - distanceKm × 2) via Haversine formula
    2Rating+25(rating / 5) × 25
    3Reliability / On-Time Score+20(reliabilityScore / 100) × 20
    4Cancellation Rate−15(cancellationRate / 15) × 15 penalty
    5Availability / Capacity+10(availableSlots / capacity) × 10
    6Budget Sensitivity+10+10 if user is budget-sensitive AND baseRate < 1000
    7Specialization Match+15+15 if provider's specializations[] includes requested category
    8Review Recency+10≤7 days=10, ≤30 days=7, ≤60 days=4, older=1
    9Risk Score−15(riskScore / 100) × 15 penalty
    10Complexity Fit+10Complex jobs: +10 if experienceYears≥3 AND totalBookings≥100; +5 if experienceYears≥2
    Example scoring trace (from live request sample-trace-001):

    Provider: Farhan Ansari (score: 99.5/115 max)
      +17.0  distance (6.5 km)
      +23.5  rating (4.7/5)
      +17.8  reliability (89%)
      -3.0   cancellation rate (3%)
      +5.0   availability (2/5 slots used)
      +10.0  budget-friendly rate (Rs.871)
      +15.0  specialization match (ac-repair, appliance-repair)
      +7.0   review recency (17 days ago)
      -2.9   risk score (19)
      +5.0   complexity fit (310 bookings, 5 years)

    Why Farhan over a closer provider? Specialization match (+15) + recency (+7) + low risk outweigh the distance penalty, exactly as the requirements specify.

    Antigravity Agent Workflow

    System Prompt Design (intentAgent.js)


    The Gemini model is called with a structured JSON schema enforced via responseMimeType: "application/json". The prompt instructs it to:
  • Handle Urdu, Roman Urdu, English, code-switching, slang, misspellings

  • Output confidence_score (0.0–1.0)

  • Set clarification_needed: true if confidence < 0.7 or if location_sector is missing

  • Return clarification_question in the same language as the user's input

  • Extract: service_category, location_sector, time_preference, urgency, budget_sensitivity, complexity

    Assumptions

  • Mock Providers: 60 procedurally generated providers across Islamabad/Rawalpindi sectors. Real deployment would source from a live provider registry or Google Sheets.

  • Authentication: Hardcoded USER-123 for the demo. Production would use Firebase Auth or JWT.

  • SMS/WhatsApp: Notification text is simulated in the booking response. Production would use Twilio or Meta Cloud API.

  • Calendar Integration: calendarEntry object generated but not pushed to Google Calendar. Production would use the Google Calendar API.

  • Payment: No real payment processing. Pricing is quoted; payment integration would use JazzCash / EasyPaisa APIs.

  • SQLite Persistence: sql.js stores data in memory per Cloud Run instance (resets on cold start). Production would use Cloud SQL or Firestore.

  • Surge Pricing: Surge multipliers are based on simulated Pakistani service demand patterns, not live telemetry.

    Privacy Note

  • No real user data is collected. The userId is hardcoded as USER-123 for demo purposes.

  • Provider data is entirely synthetic — no real names, phone numbers, or locations are from actual individuals.

  • The Gemini API processes user input text only to extract structured intent. No input text is stored beyond the active request session.

  • GPS coordinates captured from the mobile device are used only to find the nearest sector (Haversine formula). They are never sent to the server or stored.

  • All SQLite data is ephemeral (resets on Cloud Run cold start).

    • Limitations

  • Single LLM Call Architecture: Currently one Gemini call per request. Complex multi-turn conversations (follow-up questions) would need session state management.

  • No Real-Time Provider Availability: jobsToday is static per deployment. Real system needs live provider status updates.

  • Geographic Coverage: Only 27 Islamabad/Rawalpindi sectors + 4 Karachi sectors. Other Pakistani cities not covered.

  • No Voice Input: The microphone button in the UI is a placeholder. Full integration would use Whisper or Google Speech-to-Text.

  • Concurrency at Scale: providers.json file-level locking for reputation updates is not suitable for high concurrency. Production needs atomic DB updates.

  • SQLite Ephemeral: Booking history resets on each Cloud Run instance restart (cold start). Production would use Cloud SQL.

  • Surge Pricing: Based on simulated demand curves, not real market data.

    Problem Statement

    Pakistan's informal service economy (electricians, plumbers, AC technicians, cleaners) has no structured booking system. Users rely on word-of-mouth or WhatsApp. Providers get no-shows. There is no quality tracking, no fair pricing, and no multilingual interface for Urdu-speaking users.

    Goal: Build a fully autonomous, agentic AI system that handles the complete lifecycle of a service request — from multilingual intent parsing to provider matching, pricing, booking, follow-up, and dispute resolution.

    Architecture Decision

    Why Multi-Agent?


    A single LLM call cannot reliably handle all concerns (NLP + business logic + pricing + persistence + reputation). A multi-agent architecture separates concerns, makes each step auditable, and allows independent improvement of each agent.

    Why Gemini 2.5 Flash?


  • Structured JSON output via responseMimeType: "application/json"

  • Cost-efficient (~$0.00005/request)

  • Handles multilingual input natively

  • Fast enough for real-time mobile UX (~3.5s end-to-end)

    • Why Cloud Run?


  • Serverless — scales to zero (no idle cost)

  • Stateless — matches the sql.js in-memory DB model

  • Docker-native — same Dockerfile works locally and in production

    • Agent Design

    Agent 1 — Intent Agent (Gemini 2.5 Flash)


    Responsibility: Parse natural language input in any language/script
    Input: Raw user text (Urdu, Roman Urdu, English, mixed)
    Output: Structured JSON with service_category, location_sector, urgency, complexity, budget_sensitivity, confidence_score, clarification_needed
    Fallback: If confidence_score < 0.7 OR location missing → return clarification_question in user's language

    Agent 2 — Matching Agent (Deterministic, 10-Factor)


    Responsibility: Rank all available providers for the requested service
    Factors:
  • Distance (Haversine, max 30 pts)

  • Rating (max 25 pts)

  • Reliability score (max 20 pts)

  • Cancellation rate penalty (max -15 pts)

  • Capacity/availability (max 10 pts)

  • Budget sensitivity (max 10 pts)

  • Specialization match (max 15 pts)

  • Review recency (max 10 pts)

  • Risk score penalty (max -15 pts)

  • Complexity fit (max 10 pts)

    • Output: Top 3 providers with full per-factor reasoning

    Agent 3 — Pricing Agent (Deterministic)


    Responsibility: Calculate transparent, dynamic price
    Formula: (baseRate × complexityMultiplier + urgencyPremium + distanceCost) × surgeMultiplier − loyaltyDiscount
    Surge: 1.2× during peak hours (8–10am, 5–9pm PKT)
    Loyalty: 5% off for users with 3+ past bookings
    Budget Alternative: Quoted when budget_sensitivity=high

    Agent 4 — Booking Agent (SQLite + AsyncStorage)


    Responsibility: Confirm booking with travel-time buffer, handle capacity limits
    Logic:
  • Check jobsToday < capacity before confirming

  • Add travel-time buffer to scheduled slot

  • Insert to SQLite + cache to mobile AsyncStorage

  • If full → return WAITLISTED with 3 alternate slots

  • Generate SMS receipt text + calendar entry metadata

    • Agent 5 — Follow-Up Agent


    Responsibility: Manage service lifecycle after booking
    States: EN_ROUTE → IN_PROGRESS → COMPLETED
    On COMPLETED: Run 6-step quality checklist, update provider reliabilityScore +1, prompt user rating

    Agent 6 — Dispute Agent


    Responsibility: Auto-resolve customer complaints
    Types & Resolutions:
  • NO_SHOW → Cancel charge-free + Rs.500 credit + reliability -8

  • QUALITY_ISSUE → 30% refund + supervisor rework + reliability -4

  • PRICE_MISMATCH → Hold payment + human escalation + reliability -2

  • OVERRUN → Case escalation + risk score +5

    • Persistence: All penalty changes written to providers.json → affect future matching

    Agent 7 — Workload Agent


    Responsibility: System-wide demand forecasting and provider optimization
    Output: Per-category utilization, 24-hour demand forecast, recommended low-demand booking slots

    Orchestrator — Master Controller


    Responsibility: Sequence all agents, handle errors, return unified response
    Error Handling:
  • QUOTA_EXCEEDED → User-friendly quota message

  • MODEL_NOT_FOUND → Config error message

  • INVALID_API_KEY → Support escalation message

  • NO_PROVIDERS → Alternate slots + waitlist suggestion
